# i18n — Adding a string / adding a locale

Tokenova maintains KO/EN dictionaries in two places — frontend + backend. Any addition must stay in sync across both.

## Quick flow

```
Adding a new UI string:
  1. Add the key + Korean text to src/i18n/locales/ko.ts
  2. Add the same key + English to src/i18n/locales/en.ts (TS compiler flags omissions)
  3. Call t("nav.your_key") in code
  4. (For static HTML)  <span data-i18n="nav.your_key">fallback</span>
```

## Frontend dictionary structure

`src/i18n/locales/ko.ts` is the master. The TypeScript type of that object becomes the contract every other locale must satisfy:

```typescript
// ko.ts
export const ko = {
  nav: {
    today: "오늘",
    codex: "도감",
    // ... new key:
    settings: "설정",
  },
  // ...
};

// en.ts
import type { Dictionary } from "..";
export const en: Dictionary = {
  nav: {
    today: "Today",
    codex: "Codex",
    settings: "Settings",  // ← omission = compile error
  },
  // ...
};
```

Run `npx tsc --noEmit` to verify type alignment. CI runs the same check.

## The `t()` function

```typescript
import { t } from "../i18n";

// Static string
t("nav.today")                        // → "오늘" or "Today"

// {placeholder} interpolation
t("today.draw_bar.connected_n", { count: 3 })
// → "별 3개 연결됨" or "3 stars connected"
```

Placeholders use `{name}` syntax. Missing params or missing keys leave the placeholder text visible — easy to spot in dev.

## Static HTML strings

Mark with `data-i18n` and `applyDomI18n()` fills it in:

```html
<button data-i18n="today.draw_bar.cancel">Cancel</button>
<input data-i18n-attr-placeholder="today.cluster_name_placeholder" />
<span data-i18n-attr-title="today.tokens_label">Tokens</span>
```

The original text acts as a fallback — `applyDomI18n` overwrites it on init and on every locale change.

## Dynamic + interpolated

When setting textContent / innerHTML directly:

```typescript
const $label = document.getElementById("draw-bar-label");
if ($label) $label.innerHTML = t("today.draw_bar.connected_n", { count });
```

## Backend (Rust) dictionary

Notifications / tray text / backend-emitted strings live in `src-tauri/src/i18n.rs`:

```rust
pub fn planet_rarity_title(locale: Locale, rarity: Rarity) -> &'static str {
    match (locale, rarity) {
        (Locale::Ko, Rarity::Mythic) => "신화 행성 발견!",
        (Locale::Ko, Rarity::Legendary) => "전설 행성 발견!",
        // ...
        (Locale::En, Rarity::Mythic) => "Mythic planet discovered!",
        (Locale::En, Rarity::Legendary) => "Legendary planet discovered!",
        // ...
    }
}
```

To add a new notification string:
1. Add a new case to the Rust function signature (KO + EN both).
2. At the call site, `let locale = crate::i18n::current_locale(&self.db);` and pass it through.

`current_locale` looks up the `locale` key in the `settings` table on each call — notifications are infrequent, no caching needed.

## Locale state + change

```typescript
import { getLocale, setLocale, subscribeLocale } from "../i18n";

getLocale()           // "ko" | "en"
await setLocale("en") // persist + applyDomI18n + notify subscribers

subscribeLocale((newLocale) => {
  // this view needs to redraw its dynamic strings
  refresh();
});
```

Polling views (like today.ts's 3-second poll) pick up the new locale on the next tick automatically. Views that need immediate refresh (like settings.ts's update banner state) should `subscribeLocale`.

## Adding a new locale (e.g. Japanese)

1. **Extend the type**: `src/i18n/index.ts`
   ```typescript
   export type Locale = "ko" | "en" | "ja";
   ```
2. **Create the dictionary**: `src/i18n/locales/ja.ts` — copy `en.ts` as a starting point, translate every value. The `Dictionary` type catches missing keys.
3. **Register in the DICTS map**: `src/i18n/index.ts`
   ```typescript
   import { ja } from "./locales/ja";
   const DICTS: Record<Locale, Dictionary> = { ko, en, ja: ja as Dictionary };
   ```
4. **Extend persistence validation**: `src-tauri/src/commands.rs::set_locale`
   ```rust
   if value != "ko" && value != "en" && value != "ja" {
       return Err(format!("unsupported locale: {value}"));
   }
   ```
5. **Extend detectSystemLocale**: `src/i18n/index.ts`
   ```typescript
   if (l.toLowerCase().startsWith("ja")) return "ja";
   ```
6. **Add a Settings UI button**: `index.html`
   ```html
   <button class="settings-lang-btn" data-locale="ja" data-i18n="settings.language.ja">日本語</button>
   ```
7. **Extend the backend i18n**: `src-tauri/src/i18n.rs` — add `Locale::Ja` to the enum + every function's match arms.

## Validation

```bash
# TS dictionary consistency
npx tsc --noEmit

# Rust dictionary consistency (every enum variant covered)
cargo check
```

Rust enforces exhaustiveness — missing a match arm is a compile error. Safe.

## Priority / fallback

1. User-specified setting (the `locale` key in DB)
2. `navigator.languages` heuristic (system OS locale)
3. Final fallback: Korean (the master dictionary)

When a specific key is missing:
- Not in current locale → try master (ko)
- Not in ko either → return the key string itself (instantly visible in dev)
